# AGENTS.md

## Project
Small Go HTTP server serving iCalendar feeds. Single module: `example.com/ical`.

## Commands
```sh
go run main.go          # run (default port 8080, override with PORT env var)
go build -o ical-server .
go test ./...
go vet ./...
gofmt -w .
sqlc generate           # regenerate db/ after editing query.sql or schema.sql
```

No Makefile, no CI, no test files yet.

## Package layout
| Path | Role |
|---|---|
| `main.go` | Entry point; embeds `schema.sql`, wires DB + routes |
| `handlers/` | HTTP handlers |
| `render/` | RFC 5545 iCalendar string builder |
| `event/` | Domain struct + CRUD (uses raw SQL against a global `db.DB`) |
| `db/` | **sqlc-generated** — do not edit by hand |
| `static/` | Frontend HTML served at `/` |

## Codegen — critical
`db/query.sql.go`, `db/models.go`, `db/db.go` are auto-generated by sqlc v1.31.1.  
**Edit `query.sql` or `schema.sql`, then run `sqlc generate`.** Never edit `db/` directly.

`schema.sql` is embedded into the binary at compile time via `//go:embed schema.sql` in `main.go`.

## Known issues (work-in-progress)
- DB is **in-memory only** (`:memory:`); all data is lost on restart.
- `event/event.go` references a raw global `db.DB` that is not initialized anywhere — this code path is broken.
- `main.go` uses `fmt.Sprintf("0.0.0.0:%d", port)` with `%d` but `port` is a string.
- `query.sql` UPDATE sets `location = ?` twice, producing a `Location_2` field in generated code.
- The ICS calendar handler currently returns an empty event list; the event creation handler has a broken body-read pattern.

## Architecture note
Two overlapping DB access patterns coexist: the sqlc `Queries` struct path (used in `main.go`) and the raw global `db.DB` path (used in `event/event.go`). Prefer the sqlc path for new code.
